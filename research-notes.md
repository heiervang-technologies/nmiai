## 1. What Is NM i AI?

NM i AI (also referred to as NM i KI / Norgesmesterskapet i kunstig intelligens) is Norway's national AI championship — a four-day, task-oriented AI competition. It is the largest AI competition in Norway, open to everyone: students, professionals, hobbyists, public sector workers, and retirees.

The competition emphasizes **execution, problem-solving, and technical decision-making under real conditions**. Teams receive three simultaneous challenges and submit solutions through an automated API scoring system with real-time leaderboard feedback.

## 10. NorgesGruppen Object Detection Challenge - Full Specification

### Challenge Overview
- **Task:** Detect and classify grocery products on store shelves
- **Sponsor:** NorgesGruppen Data (Norway's largest grocery retailer group)
- **Type:** Code upload (ZIP file) - not an API endpoint
- **Scoring weight in overall competition:** 25% (average of 4 tasks: Grocery Bot, Tripletex, Object Detection, Astar Island)

### Training Data
- **COCO dataset:** `NM_NGD_coco_dataset.zip` (864 MB)
  - 248 shelf images
  - ~22,700 bounding box annotations
  - 356 product categories (IDs 0-355)
  - Standard COCO JSON annotation format
- **Product reference images:** `NM_NGD_product_images.zip` (60 MB)
  - 327 products with multi-angle photos
  - Studio-quality reference images of individual products

### Submission Format
- **Upload:** ZIP file, max 420 MB
- **Entry point:** `run.py` at ZIP root (critical - must be at root level)
- **Input:** `/data/images/*.jpg` directory of test shelf images
- **Output:** JSON array with predictions containing:
  - `bbox` (bounding box coordinates)
  - `category_id` (product category 0-355)
- **Security restrictions:** No `os`, `subprocess`, or `socket` imports allowed
- **Model format compatibility:** PyTorch native, ONNX export, SafeTensors
- **Daily quota:** 3 submissions per day
- **In-flight limit:** 2 concurrent submissions

### Sandbox Runtime Environment
- **GPU:** NVIDIA L4 (24 GB VRAM)
- **Pre-installed software:**
  - PyTorch 2.6.0
  - YOLOv8 (Ultralytics)
  - ONNX Runtime
  - OpenCV
- **Timeout:** 300 seconds per submission
- **Max model weight size:** 420 MB (same as zip limit)

### Scoring Metric
- **70% detection mAP** (at IoU >= 0.5) - how well boxes localize objects
- **30% classification mAP** (at IoU >= 0.5) - how well categories are assigned
- This is a **weighted split** - detection accuracy matters more than classification
- IoU threshold is 0.5 (relatively lenient - boxes only need 50% overlap)

### Key Constraints Summary
| Constraint | Value |
|-----------|-------|
| GPU | NVIDIA L4 (24GB VRAM) |
| Timeout | 300 seconds |
| Max ZIP size | 420 MB |
| Categories | 356 (IDs 0-355) |
| Training images | 248 |
| Training annotations | ~22,700 |
| Reference products | 327 (multi-angle) |
| Daily submissions | 3 |
| Concurrent submissions | 2 |

## 11. Model Architecture Comparison for This Challenge

### Key Decision Factors
- 420 MB zip limit (model weights + code + dependencies)
- 300-second inference timeout on L4 GPU
- 356 classes with only 248 training images (~22.7k annotations)
- mAP@0.5 (lenient IoU) with 70/30 detection/classification split
- Pre-installed: PyTorch 2.6, YOLOv8, ONNX Runtime

### YOLO Family

#### YOLO11 (Ultralytics, pre-installed as YOLOv8)

|---------|-----------------|------------|-----------------|---------------|
| YOLO11n | 39.5 | 2.6 | ~5 MB | 1.5 |
| YOLO11s | 47.0 | 9.4 | ~19 MB | 2.5 |
| YOLO11m | 51.5 | 20.1 | ~40 MB | 4.7 |
| YOLO11l | 53.4 | 25.3 | ~50 MB | 6.2 |
| YOLO11x | 54.7 | 56.9 | ~115 MB | 11.3 |

**Pros:**
- Pre-installed in sandbox (YOLOv8/Ultralytics framework)
- Very fast inference - plenty of time budget
- Mature fine-tuning pipeline with COCO format
- Strong augmentation built-in (mosaic, mixup, copy-paste)
- All variants fit easily within 420 MB limit
- Well-suited for mAP@0.5 (good recall)

**Cons:**
- Lower absolute mAP than transformer models
- May struggle with 356 fine-grained categories

#### YOLO26 (Ultralytics, January 2026)

| Variant | mAP50-95 (COCO) | Params (M) | T4 Speed (ms) |
|---------|-----------------|------------|---------------|
| YOLO26n | 40.9 | 2.4 | 1.7 |

**Pros:**
- Latest YOLO - 1-3% mAP improvement over YOLO11
- NMS-free end-to-end inference

- May need to install ultralytics update in zip

**Cons:**
- May not be pre-installed in sandbox
- Need to verify compatibility with PyTorch 2.6

### RF-DETR (Roboflow, ICLR 2026)

| Variant | AP50 (COCO) | AP50-95 (COCO) | Params (M) | T4 Speed (ms) |
|---------|-------------|----------------|------------|---------------|
| Nano | 67.6 | 48.4 | 30.5 | 2.3 |
| Small | 72.1 | 53.0 | 32.1 | 3.5 |
| Medium | 73.6 | 54.7 | 33.7 | 4.4 |
| Large | 75.1 | 56.5 | 33.9 | 6.8 |

**Pros:**
- SOTA on COCO, first real-time model >60 AP
- DINOv2 backbone (excellent feature extraction)
- Specifically designed for fine-tuning - converges in fewer epochs on small datasets
- Strong AP50 scores (what matters here)
- NMS-free architecture
- Base model ~30M params (~120 MB weights) - fits in 420 MB

**Cons:**
- Not pre-installed - must bundle in zip
- rfdetr package + weights add to zip size
- Large variants (XL, 2XL) are 126M params - may exceed zip limit
- Requires Python >= 3.10

### Florence-2 (Microsoft)

**Pros:**
- Strong zero-shot and few-shot capabilities
- Can leverage text descriptions of products
- Fine-tunable with LoRA for efficiency
- Good for domain adaptation with limited data

**Cons:**
- Not optimized for speed in detection tasks
- Model size may be challenging within 420 MB
- More complex inference pipeline
- Not pre-installed

### Grounding DINO

**Pros:**
- 52.5 AP zero-shot on COCO (no training needed!)
- 63.0 AP when fine-tuned on COCO
- Can use product names as text prompts
- Open-vocabulary: can detect unseen categories
- Fine-tuning with Supervised Prompt Tuning (SPT) for custom categories

**Cons:**
- Large model (Swin-T/BERT-base backbone)
- Slower inference than YOLO/RF-DETR
- Complex setup - likely exceeds 420 MB
- 300s timeout may be tight for many images

### DE-ViT (Few-Shot with DINOv2)

**Pros:**
- Designed exactly for reference-image-based detection
- Uses DINOv2 backbone, learns from example images (not text)
- Outperforms open-vocabulary SOTA by 6.9 AP50
- 50 AP50 on novel classes with just reference images
- Perfect for 327 product reference images scenario

**Cons:**
- Research code, less production-ready
- May be difficult to fit + run within constraints
- Complex pipeline

## 12. Recommended Approach Strategy

### Primary Strategy: Fine-tune YOLO11x or YOLO26x

**Rationale:**
1. YOLO is pre-installed in the sandbox (guaranteed compatibility)
2. 248 images with 22.7k annotations is actually decent for fine-tuning
3. YOLO excels at mAP@0.5 (lenient IoU threshold)
4. The x variant (~115 MB) fits easily in 420 MB with room for code
5. Fast inference leaves time budget for TTA
6. 356 classes is unusual but YOLO handles large class counts

**Training Recipe:**
- Start from COCO-pretrained YOLO11x or YOLO26x
- Fine-tune on the 248 shelf images for 100-300 epochs
- Heavy augmentation: mosaic, mixup, copy-paste, flips, scale, rotation
- Use product reference images for copy-paste augmentation onto shelf backgrounds
- Lower confidence threshold (0.001-0.01) for mAP evaluation
- Tune NMS IoU threshold empirically

### Alternative Strategy: RF-DETR Medium/Large

**Rationale:**
1. Higher mAP than YOLO, especially on small/crowded objects
2. DINOv2 backbone has better feature representations
3. Designed for fine-tuning on small custom datasets
4. Medium model (~34M params, ~140 MB) fits in 420 MB

**Risk:** Must bundle rfdetr package in zip, compatibility with sandbox

### Hybrid Strategy: YOLO Detection + DINOv2 Classification

**Rationale:** Given the 70/30 detection/classification split:
1. Use YOLO for detection (find bounding boxes) - optimizes the 70%
2. Use DINOv2 embeddings to match detected crops against product reference images - optimizes the 30%
3. Product reference images (327 products, multi-angle) are perfect for embedding-based matching
4. DINOv2 embeddings show strong discriminative power for similar objects

**Implementation:**
- YOLO detects all products (possibly class-agnostic or coarse classes)
- Crop each detection from image
- Extract DINOv2 embedding of crop
- Compare against pre-computed embeddings of 327 reference products
- Assign nearest-neighbor category_id

## 13. mAP@0.5 Optimization Strategies

### Why mAP@0.5 Favors Certain Approaches

- IoU 0.5 is lenient: bounding boxes only need 50% overlap
- Recall matters more than precise localization
- Loose/generous boxes are acceptable (padding is fine)
- Focus on not missing objects rather than pixel-perfect boxes

### Confidence Threshold
- For evaluation, use very low confidence threshold (0.001)
- This maximizes recall at the cost of more false positives
- The precision-recall curve calculation handles the trade-off
- At inference time for submission: keep threshold low

### NMS Strategies
- Standard NMS: lower threshold (0.4-0.5) for dense grocery shelves
- Soft-NMS: reduces scores instead of eliminating, better for overlapping products
- DIoU-NMS: considers center distance, good for similar adjacent products
- NMS-free models (YOLO26, RF-DETR): avoid NMS issues entirely

### Test-Time Augmentation (TTA)
- Multi-scale inference: test at 2-3 scales (e.g., 640, 832, 1024)
- Horizontal flip augmentation
- Merge predictions with NMS
- Typical improvement: +1-2% mAP
- Cost: 2-3x inference time (still within 300s on L4)

### Input Resolution
- Higher resolution helps detect small products on shelves
- YOLO default 640 may be too low for shelf images
- Consider 1024 or 1280 input size
- L4 with 24GB VRAM can handle high-res inference

### Post-Processing
- Weighted Boxes Fusion (WBF) instead of NMS for multi-model ensembles
- Category-specific confidence calibration
- Remove detections below minimum reasonable product size

## 14. Data Augmentation for Limited Training Data (248 images)

### Standard Augmentations
- **Mosaic:** Combines 4 images into 1 - effectively 4x data
- **MixUp:** Alpha-blends two images and labels
- **Random flip/rotation/scale/translate**
- **Color jittering:** Brightness, contrast, saturation, hue
- **Random erasing:** Simulates occlusion

### Copy-Paste Augmentation (Critical for This Challenge)
- Cut product segments from reference images
- Paste onto shelf backgrounds
- Automatically generates new training annotations
- 2x data-efficiency improvement demonstrated on COCO
- +9.7% improvement on small objects
- Product reference images (327 products, multi-angle) are ideal source material

### Synthetic Data Generation
- Compose product reference images onto shelf layouts
- Vary scale, rotation, lighting
- Use geometric layout augmentation
- Generative harmonization for realistic compositing

### Class Imbalance Handling
- 356 categories across ~22.7k annotations = ~64 annotations per class average
- Some classes likely have very few examples
- Oversample rare classes
- Focal loss to handle class imbalance
- Category-aware sampling strategies

## 15. Few-Shot and Reference Image Approaches

### Using 327 Product Reference Images

The reference images provide multi-angle studio photos of individual products. Key approaches:

#### A. Copy-Paste Training Augmentation
- Segment products from reference photos (easy - studio backgrounds)
- Paste onto shelf training images at various positions/scales
- Generate new COCO annotations automatically
- Most practical approach - directly augments training data

#### B. DINOv2 Embedding Matching (Post-Detection Classification)
- Pre-compute DINOv2 embeddings for all 327 reference products
- After YOLO detection, crop each detection
- Compute DINOv2 embedding of crop
- Nearest-neighbor match against reference embeddings
- Handles products not well-represented in training data
- DINOv2 shows strong discriminative power for similar objects

#### C. DE-ViT (Open-Set Detection)
- Purpose-built for few-shot detection with reference images
- 50 AP50 on novel classes
- Builds class prototypes from example images
- Binary classification approach (is/isn't each class)
- Could detect products never seen in shelf training images

#### D. Grounding DINO with Product Names
- Use product names as text prompts
- Zero-shot detection capability (52.5 AP on COCO)
- Can be fine-tuned with Supervised Prompt Tuning
- Risk: product names may not be descriptive enough for visual matching

### Recommended Combination
1. Primary: Fine-tune YOLO on augmented dataset (copy-paste from reference images)
2. Secondary: DINOv2 embedding re-classification for hard/rare categories
3. This targets both the 70% detection and 30% classification scoring
